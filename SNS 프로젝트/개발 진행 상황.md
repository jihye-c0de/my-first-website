## 2. 핵심 기능 구현 현황 (기획서 4번 기준)

**필수 기능**
- 프로필 생성 및 관리 — ✅ (프로필 사진 업로드 포함)
- 게시물 작성/수정/삭제 — ✅ (헤더의 점세개 메뉴로 통합)
- 팔로우/언팔로우 — ✅
- 좋아요/댓글/저장 — ✅ **좋아요 영속화 완료**: `ate_likes` 테이블 + 트리거로 실제 좋아요 여부를 저장해 새로고침해도 상태 유지, 유저당 게시물 하나에 한 번만 좋아요(다시 누르면 취소, 인스타그램 방식). 댓글은 아이콘 클릭 시 카드 안에서 바로 인라인 작성(상세 페이지 이동 없음)

**부가 기능**
- 네이버 지도 링크 연결 — ✅ (공유 텍스트를 통째로 붙여넣어도 URL만 자동 추출하도록 방어 처리)
- 공유하기 — ✅ (Web Share API, 미지원 브라우저는 클립보드 복사)
- 1:1 다이렉트 메시지 — ❌ 미구현 (사용자와 합의하여 이번 범위에서 제외)
- 저장한 피드 컬렉션 — ⚠️ **절반만 구현**. `src/lib/social.js`에 `fetchSavedPosts()` 함수는 있고 저장/취소 동작은 되지만, "저장한 게시물 모아보기" 화면(전용 페이지)이 아직 없음 — **다음 작업 1순위**
- 사진 자동 배경 제거(누끼) — ✅ 완료. 배경 제거 모델/wasm을 우리 저장소에 자체 호스팅(`public/bg-removal`)해 외부 CDN 의존성 제거, 가벼운 양자화 모델(isnet_quint8) 사용, 20초 타임아웃 시 원본으로 자동 대체
- 원본/누끼 사진 분리 — ✅ 완료(추가 기능). `original_image_url` 컬럼 추가: 탐색·마이페이지 게시물 그리드는 원본 사진(cover), 홈 피드·상세·캘린더는 누끼 사진을 사용. 이 기능 적용 이전에 작성된 게시물은 원본이 없어 누끼 사진으로 대체 표시됨

## 3. DB 구현 현황

핵심 테이블: `ate_users` / `ate_posts` / `ate_comments` / `ate_follows` / `ate_saved_posts` / **`ate_likes`(신규)**. RLS 전체 적용.

`ate_posts` 주요 컬럼 추가 이력:
- `is_cutout` — 배경 제거 성공 여부(그리드/캘린더에서 프레임 유무 결정)
- `original_image_url` — 누끼 처리 전 원본 사진 URL
- `is_calendar_only` — 캘린더 전용 기록 여부(피드/탐색/마이페이지 게시물 탭 제외)

Storage 버킷: `post-images`, `avatars`(신규, 프로필 사진), `bg-removal`(신규, 배경 제거 모델 정적 파일 — Storage 버킷이 아니라 앱 자체의 `public/` 정적 파일임).

`ate_adjust_post_likes` RPC는 `ate_likes` 테이블 도입 후 **삭제**(개인별 좋아요 추적 없이 카운트만 조정하던 방식 폐기). `get_advisors` 보안 점검 완료(트리거 전용 함수 RPC 노출 차단 등).

## 4. 개발 중 발견/수정한 버그 (다음에 비슷한 이슈 만나면 참고)

1. **PostgREST 조인 모호성**: 조인 테이블(`ate_saved_posts`)이 생기면 관련 테이블 간 임베드 select가 여러 경로로 해석되어 조용히 실패(빈 목록으로 위장)할 수 있음. `ate_users!ate_posts_user_id_fkey(...)` 형태로 관계를 명시해서 해결. **새 조인 테이블을 추가할 때마다 기존 임베드 select들이 모호해지지 않는지 확인 필요.**
2. **마이페이지 무한 로딩**: 예외 발생 시 `setLoading(false)`가 실행되지 않아 로딩이 영구히 멈추던 버그. try/catch/finally로 수정.
3. **배경 제거 무한 대기**: WASM 추론이 응답 없이 멈추면 게시 버튼이 계속 비활성화됐음. 20초 타임아웃 후 원본 사진으로 자동 대체.
4. **좋아요/저장 클로저 버그**: `handleSave`에서 `checkSaved()` 직후 `if (saved)`를 참조했는데, setState는 비동기라 그 시점엔 이전 상태를 읽고 있었음. 마운트 시 좋아요/저장 상태를 미리 조회하고, 토글은 `nextValue`를 미리 계산해 사용하도록 수정.
5. **프로필 사진 업로드 RLS 실패**: storage upload에 `upsert: true`를 쓰면 새 파일인데도 update 경로로 RLS를 검사해 "new row violates row-level security policy" 오류 발생. 매번 고유 파일명을 쓰므로 upsert 제거로 해결.
6. **네이버 지도 링크 404**: 네이버 지도 앱의 "공유" 텍스트(`[네이버지도] 상호명 주소 https://...`)를 통째로 붙여넣으면 스킴 없는 문자열이 그대로 저장되어 상대경로로 해석되며 404. 텍스트에서 실제 URL만 추출(`utils/url.js`)하도록 저장 시점 + 렌더링 시점 양쪽에서 방어.
7. **배경 제거가 외부 CDN에 의존**: `@imgly/background-removal` 기본값이 매 요청마다 IMG.LY의 `staticimgly.com`에서 모델(수십MB)을 받아오는 구조라 네트워크 상태에 따라 느리거나 실패. 필요한 모델/wasm 파일만 골라 우리 저장소(`public/bg-removal`)에 포함하고 `publicPath`를 우리 도메인으로 지정해 해결.
8. **모바일에서 갤러리 선택 불가**: 파일 입력에 `capture="environment"`가 있으면 일부 모바일 브라우저가 갤러리 선택 없이 카메라로 바로 넘어감. 해당 속성 제거.
9. **탐색에서 클릭한 게시물이 아닌 다른 게시물이 먼저 보임**: 클릭한 게시물로 `scrollIntoView`하는 방식이었는데, 각 게시물이 비동기 로딩되며 높이가 계속 바뀌어 스크롤 위치가 다시 맨 위로 밀림. 스크롤 대신 클릭한 게시물이 항상 목록 맨 앞에 오도록 순서 자체를 재배열하는 방식으로 수정.
10. **중첩 flex로 아이콘이 완전히 중앙 정렬되지 않음**: 작성 선택 카드에서 `ButtonBase` 안에 `Stack`(alignItems=center)을 중첩해도 아이콘이 시각적으로 중앙에 오지 않았음. 중첩 flex 조합 대신 `textAlign: 'center'` + 고정폭 요소에 `mx: 'auto'`를 쓰는 block 기반 정렬로 교체해 해결. **아이콘/원형 배지처럼 고정 크기 요소를 중앙 정렬할 땐 중첩 flex보다 mx: auto 방식이 더 안정적임.**

## 5. 기획서에 없지만 추가로 구현한 것

- 데모/시드 데이터: 가짜 유저 20명 + 게시물 21개 + 댓글/팔로우 관계, 유저별 기본 아바타(DiceBear, 일부는 공통 기본값 유지)
- GitHub Actions 기반 자동 배포 파이프라인
- 게시물 상세 "이어서 스크롤 보기" + 댓글 인라인 작성 + 게시물 헤더 점세개 메뉴(수정/삭제 통합)
- 앱 콘텐츠 영역(하단 탭 너비)과 바깥 여백을 배경색으로 구분
- 원본 사진 / 누끼 사진 분리 저장 및 화면별 다른 사진 사용
- 캘린더 전용 기록(게시물로 노출되지 않는 개인 다이어리성 기록) + 팝업 미리보기
- 배경 제거 모델 자체 호스팅(외부 CDN 의존성 제거)
- 네이버 지도 공유 텍스트 자동 URL 추출

## 6. 다음에 이어서 할 만한 작업 (우선순위 순)

1. **저장한 게시물 모아보기 페이지** — `fetchSavedPosts()`는 이미 있음, 마이페이지에 탭 추가하거나 별도 라우트만 연결하면 됨
2. 소셜 로그인(카카오/구글) — Supabase 대시보드에서 OAuth 클라이언트 등록 필요 (수동 작업)
3. 실제 네이버 지도 임베드 — 네이버 클라우드 플랫폼 Maps API Client ID 발급 필요
4. 1:1 다이렉트 메시지 — 별도 실시간 테이블/구독 로직 필요, 개발 범위 큼
5. 캘린더 전용 기록 수정/삭제 기능 — 현재는 생성만 가능, 편집·삭제 UI 없음

## 7. UI 다듬기 이력 (참고용)

- 작성 선택 화면(`/create`)은 여러 번 반복 수정됨: 좌우 풀스크린 분할 → 참고 이미지 기반 둥근 카드 2열 그리드(space-between, 라운드 30px) → 아이콘 중앙 정렬 버그 수정. 최종 형태는 카드 2개가 화면 중앙에 나란히, 각 카드 안에 원형 아이콘 배지 + 제목 + 설명이 전부 중앙 정렬.
- 게시물 카드(피드/상세)는 인스타그램 스타일에 맞춰 여러 차례 조정: 사진 크기(4:5, 최대 높이 480px, 프레임 제거) → 위치 태그 헤더 이동 → 댓글 인라인 통합 → 카테고리 칩 제거.
