## 2. 핵심 기능 구현 현황 (기획서 4번 기준)

**필수 기능**
- 프로필 생성 및 관리 — ✅
- 게시물 작성/수정/삭제 — ✅
- 팔로우/언팔로우 — ✅
- 좋아요/댓글/저장 — ✅ (좋아요는 DB 문서 설계대로 `likes_count` 증감만 사용, 개별 유저의 "좋아요 눌렀는지" 상태는 세션 내 클라이언트 상태로만 관리 — 새로고침하면 하트 채움 상태 초기화됨. 실제 좋아요 여부까지 영속하려면 좋아요 테이블 추가 필요)

**부가 기능**
- 네이버 지도 링크 연결 — ✅ (링크 방식)
- 공유하기 — ✅ (Web Share API, 미지원 브라우저는 클립보드 복사)
- 1:1 다이렉트 메시지 — ❌ 미구현 (사용자와 합의하여 이번 범위에서 제외)
- 저장한 피드 컬렉션 — ⚠️ **절반만 구현**. `src/lib/social.js`에 `fetchSavedPosts()` 함수는 있고 저장/취소 동작은 되지만, "저장한 게시물 모아보기" 화면(전용 페이지)이 아직 없음 — 다음에 이어서 할 작업 1순위
- 사진 자동 배경 제거(누끼) — ✅ 완료 (클라이언트 사이드 WASM, API 키 불필요) + 크기 정규화까지 추가 완료

## 3. DB 구현 현황 (데이터베이스 보조.txt 기준)

`ate_users` / `ate_posts` / `ate_comments` / `ate_follows` / `ate_saved_posts` 5개 테이블 전부 생성, RLS 전체 적용, `post-images` storage 버킷, 좋아요 증감용 RPC 함수(`ate_adjust_post_likes`) 구현 완료. `get_advisors` 보안 점검도 완료.

## 4. 개발 중 발견/수정한 버그 (다음에 비슷한 이슈 만나면 참고)

1. **PostgREST 조인 모호성**: `ate_saved_posts` 테이블이 생기면서 `ate_posts`↔`ate_users` 사이에 직접 FK 경로와 `ate_saved_posts`를 통한 간접 경로가 동시에 생겨, `ate_users(...)` 임베드 select가 전부 조용히 실패(빈 목록으로 위장)했음. `ate_users!ate_posts_user_id_fkey(...)` 형태로 관계를 명시해서 해결. **새 조인 테이블을 추가할 때마다 기존 임베드 select들이 모호해지지 않는지 확인 필요.**
2. **마이페이지 무한 로딩**: `load()` 함수 내부에서 예외가 나면 `setLoading(false)`가 실행되지 않아 로딩이 영구히 멈추던 버그. try/catch/finally로 수정.
3. **배경 제거 무한 대기**: 브라우저 환경에 따라 WASM 추론이 응답 없이 멈추면 게시 버튼이 계속 비활성화됐음. 20초 타임아웃 후 원본 사진으로 자동 대체하도록 수정.

## 6. 다음에 이어서 할 만한 작업 (우선순위 순)

1. **저장한 게시물 모아보기 페이지** — `fetchSavedPosts()`는 이미 있음, 마이페이지에 탭 추가하거나 별도 라우트만 연결하면 됨
2. 좋아요 상태 영속화 — 원한다면 `ate_likes` 테이블을 추가해 새로고침해도 좋아요 누른 상태가 유지되도록 개선 가능 (현재는 문서상 의도된 단순화)
3. 소셜 로그인(카카오/구글) — Supabase 대시보드에서 OAuth 클라이언트 등록 필요 (수동 작업)
4. 실제 네이버 지도 임베드 — 네이버 클라우드 플랫폼 Maps API Client ID 발급 필요
5. 1:1 다이렉트 메시지 — 별도 실시간 테이블/구독 로직 필요, 개발 범위 큼
